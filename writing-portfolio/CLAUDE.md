# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run start      # Local dev server with hot reload
npm run build      # Build static output to /build
npm run serve      # Serve the /build output locally
npm run clear      # Clear Docusaurus cache (use when builds behave unexpectedly)
npm run deploy     # Build and push to gh-pages branch (GitHub Pages)
```

No test suite is configured. There is no linter script in package.json.

## Architecture

This is a **Docusaurus 3.9** static site, deployed to GitHub Pages at `https://JV1996-Writer.github.io/writing-portfolio/`.

### Content areas

| Path | Purpose |
|---|---|
| `docs/` | Portfolio documents (DX Audit, GEO Guide, ShopFlow endpoint stubs as JSON) |
| `blog/` | Blog posts with `authors.yml` and `tags.yml` |
| `static/shopflow.yaml` | OpenAPI 3.0 spec for the ShopFlow demo API |
| `static/llms.txt` | GEO-optimised plain-text summary of the portfolio for AI crawlers |
| `versioned_docs/` | Docs versioned at `1.0` and `2.0` — managed by Docusaurus versioning |

### API reference rendering

The ShopFlow API spec (`static/shopflow.yaml`) is rendered **twice**:
- **Scalar** (`@scalar/docusaurus`) at `/api` — configured as a plugin in `docusaurus.config.js`
- **Redoc** (`redocusaurus` preset) at `/api-reference`

Both renderers read from the same YAML source. When modifying the spec, verify both routes still render correctly after `npm run build`.

### Custom theme components

`src/theme/ApiItem/` — swizzled Docusaurus component, overrides the default API item layout.
`src/theme/ApiItem/Layout/` — layout sub-component for the swizzled ApiItem.

Swizzled components wrap Docusaurus internals; be cautious when upgrading `@docusaurus/core` as these may need re-swizzling.

### Docs versioning

`versions.json` lists `["2.0", "1.0"]`. Versioned content lives in `versioned_docs/version-*` and `versioned_sidebars/`. The sidebar is autogenerated from the folder structure (`tutorialSidebar: [{type: 'autogenerated', dirName: '.'}]` in `sidebars.js`).

## Documentation style

- **Headings:** Sentence case only — `## How to authenticate`, not `## How To Authenticate`.
- **Voice:** Active voice throughout. Rewrite passive constructions before committing.
- **API endpoints:** Every endpoint description must include a working code example (curl or language SDK).
- **Linting:** Vale is configured with Google style and write-good (`styles/Google`, `styles/write-good`). Run Vale before committing new or edited docs.
- **Frontmatter:** All new docs must open with `title`, `description`, and `keywords` fields.
- **Answer-first (BLUF):** Lead each doc and each major section with the conclusion or key takeaway. Background and detail follow.
- **Banned words:** Do not use `utilize` (use "use"), `excellent`, `clearly`, or `significantly`.
- **Dashes:** Do not put a space before or after a dash. If a dash-separated aside is needed, use a comma instead.
- **First-person pronouns:** Do not use first-person pronouns (`I`, `we`, `my`, `our`). Write in second person or third person.
- **FAQ section:** Every new documentation page must include a FAQ section at the end.

### Frontmatter template

```yaml
---
title: Short, descriptive title
description: One sentence — what the doc covers and who it helps.
keywords: [keyword1, keyword2, keyword3]
---
```

### Deployment

Deploys to the `gh-pages` branch via:
```bash
GIT_USER=JV1996-Writer npm run deploy
```
`onBrokenLinks` and `onBrokenMarkdownLinks` are both set to `'warn'` (not `'throw'`), so broken links won't fail the build.
